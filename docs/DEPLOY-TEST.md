# Test deployment (Neon + Render)

This describes the **test** environment only. **Local development** is unchanged: use Docker Compose from the repo root ([`DEVELOPMENT.md`](./DEVELOPMENT.md)) â€” API and Postgres still run in containers there.

## Architecture

| Layer    | Provider | Role |
|----------|----------|------|
| Postgres | [Neon](https://neon.tech) | Serverless PostgreSQL; use a dedicated branch or project for test. |
| API      | [Render](https://render.com) Web Service (`runtime: python`, `rootDir: apps/api`) | `pip install .`, then `alembic upgrade head` and Uvicorn on each start â€” **no Docker image** on Render. |
| Web      | Render Static Site | Global CDN (Blueprint **must not** set `region` on static sites). Builds Vite with `VITE_API_BASE_URL` pointing at the test API. |

Infrastructure as code: [`render.yaml`](../render.yaml) at the repository root.

## 1. Neon (database)

1. Create a Neon project (free tier is enough for MVP test).
2. Create a database user and database (or use Neon defaults).
3. Copy the **connection string** for **psql** or **SQLAlchemy**. Prefer the **pooled** endpoint if Neon offers it (good for many short connections).
4. Ensure the URL includes SSL, e.g. `?sslmode=require` (Neon usually appends this).

You may paste Neonâ€™s URI as either `postgresql://â€¦` or `postgres://â€¦`. The API normalizes it to `postgresql+psycopg://â€¦` for SQLAlchemy and Alembic.

## 2. Render â€” create services from the Blueprint

1. In the Render Dashboard: **New** â†’ **Blueprint**, connect this Git repository.
2. Render reads [`render.yaml`](../render.yaml) and proposes two services: `ai-copilot-api-test` and `ai-copilot-web-test`.
3. For **ai-copilot-api-test**, set environment variables when prompted:
   - **`DATABASE_URL`**: the Neon connection string (see above).
   - **`CORS_ORIGINS`**: leave empty or temporary on first deploy; after the web service has a URL, set it to that origin (e.g. `https://ai-copilot-web-test.onrender.com`). Multiple origins: comma-separated, no spaces unless intentional (the API trims each segment).
4. Deploy the **API** first and note its public URL (e.g. `https://ai-copilot-api-test.onrender.com`).
5. For **ai-copilot-web-test**, set:
   - **`VITE_API_BASE_URL`**: the APIâ€™s public URL **without a trailing slash** (e.g. `https://ai-copilot-api-test.onrender.com`).
6. Trigger a new deploy for the static site so the bundle is built with the correct API base URL.
7. Update **`CORS_ORIGINS`** on the API to include the static site origin, then redeploy the API if needed.

`JWT_SECRET` is generated by the Blueprint unless you override it in the dashboard.

## 3. Environment reference (test)

| Variable | Service | Notes |
|----------|---------|--------|
| `DATABASE_URL` | API | Neon URI; `postgresql+psycopg` is applied automatically if missing. |
| `APP_ENV` | API | Set to `test` in `render.yaml`. |
| `JWT_SECRET` | API | Strong secret; Blueprint uses `generateValue`. |
| `CORS_ORIGINS` | API | Must include the browser origin of the static site. |
| `VITE_API_BASE_URL` | Web (build-time) | Public API URL; requires a **rebuild** when it changes. |
| `STORAGE_BACKEND` | API | `local` with `LOCAL_STORAGE_PATH=/tmp/ai-copilot-storage` (ephemeral on Render; fine for MVP test). |
| `PYTHON_VERSION` | API | `3.12.8` in `render.yaml` (must match [`apps/api`](../apps/api) `requires-python`). |

## 4. Migrations

The API **`startCommand`** runs **`alembic upgrade head`** before Uvicorn on every deploy/start, so migrations apply against Neon automatically.

## 5. Health check

Render uses **`GET /health`** for the API web service (see `render.yaml`).

## 6. Regions

The **API** web service may set `region` in [`render.yaml`](../render.yaml); if omitted, Render uses its default for new services (you can still pick a region in the dashboard). **Static sites** are edge-cached globally and do not support a Blueprint `region` field.

## 7. Production

Production hosting is not configured in this repo yet. When you add it, use separate Neon branches/projects and separate Render services or workspaces so secrets and data stay isolated from test.
